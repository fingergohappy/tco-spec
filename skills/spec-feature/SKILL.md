# spec-feature

把功能设计讨论结构化为文档，让后续实现有据可依。

## 两种模式

### 模式 A：设计讨论（对话开头触发）

如果在对话开头触发此 skill，进入设计讨论模式：

- **只讨论，不修改代码** — 这个阶段的目标是把设计想清楚，不是写代码
- 引导用户讨论：背景、核心概念、核心流程、关键机制、接口定义
- 可以阅读现有代码来理解上下文，但不做任何修改
- 当用户说"生成文档"、"输出文档"、"写下来"之类的指令时，执行文档生成

### 模式 B：直接生成（对话中途触发）

如果在对话中途触发（对话中已有设计讨论内容）：

- 直接总结已有对话，生成文档
- 跳过引导讨论环节

## 判断方式

- 对话刚开始、之前没有实质性的设计讨论 → 模式 A
- 对话中已经讨论过设计决策、有足够上下文 → 模式 B

## 文档生成步骤

1. 若 $ARGUMENTS 为空，根据对话内容自动生成一个简明的功能名称
2. 读取模板 [feature_template.md](references/feature_template.md)
3. 从对话上下文中提取设计信息，填入模板
4. 自动填充元数据:
   - `title`: $ARGUMENTS（功能名称）
   - `date`: 当前日期 (YYYY-MM-DD)
   - `status`: `draft`
   - `version`: `"1.0"`
   - `summary`: 从对话中提取 1-2 句功能目的
   - `scope`: 列出涉及的文件/模块路径
5. 输出到项目根目录下 `docs/spec/{功能名}_feature.md`
   - 如果 `docs/spec/` 目录不存在，先创建
   - 文件名使用英文，单词间用下划线连接（如 `engine_refactor_feature.md`）
   - 生成完成后告知用户文件的完整路径

## 生成规则

- 模板占位符 `{xxx}` 必须全部替换，不能残留 — 残留占位符说明信息不完整
- 代码示例用项目实际语言
- 实现规格中的文件路径必须反映实际代码路径，方便后续用 ai-kit:spec-review 对照
- 对话中未涉及的内容，根据已有代码上下文推断补充
- 如果某个 section 确实无法填充，保留标题并注明原因（"待讨论" 或 "待确认"）

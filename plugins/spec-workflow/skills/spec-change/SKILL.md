# spec-change

记录变更的全貌：改什么、为什么改、怎么改。变更文档让修改可追溯，
也让 ai-kit:spec-review 能对照检查实现是否偏离计划。

## 两种模式

### 模式 A：变更讨论（对话开头触发）

如果在对话开头触发此 skill，进入变更讨论模式：

- **只讨论，不修改代码** — 这个阶段的目标是把变更方案想清楚，不是写代码
- 引导用户讨论：变更动机、影响范围、变更清单、兼容性考虑
- 可以阅读现有代码来理解上下文，但不做任何修改
- 当用户说"生成文档"、"输出文档"、"写下来"之类的指令时，执行文档生成

### 模式 B：直接生成（对话中途触发）

如果在对话中途触发（对话中已有变更讨论内容）：

- 直接总结已有对话，生成文档
- 跳过引导讨论环节

## 判断方式

- 对话刚开始、之前没有实质性的变更讨论 → 模式 A
- 对话中已经讨论过变更方案、有足够上下文 → 模式 B

## 文档生成步骤

1. 若 $ARGUMENTS 为空，根据对话内容自动生成一个简明的变更名称
2. 读取模板 [change_template.md](references/change_template.md)
3. 从对话上下文中提取变更信息，填入模板
4. 自动填充元数据:
   - `title`: $ARGUMENTS（变更名称）
   - `type`: 根据讨论内容判断 breaking / compatible / refactor
   - `date`: 当前日期 (YYYY-MM-DD)
   - `status`: `draft`
   - `version`: `"1.0"`
   - `summary`: 从对话中提取 1-2 句变更目的
   - `scope`: 列出涉及的文件/模块路径
   - `changes`: 列出每项变更的 action/target/desc
5. 输出到项目根目录下 `docs/spec/{变更名}_change.md`
   - 如果 `docs/spec/` 目录不存在，先创建
   - 文件名使用英文，单词间用下划线连接（如 `pool_concurrency_change.md`）
   - 生成完成后告知用户文件的完整路径

## 生成规则

- 模板占位符 `{xxx}` 必须全部替换，不能残留 — 残留占位符说明信息不完整
- 代码示例用项目实际语言
- 变更清单中每个操作必须标注 [change] / [new] / [delete]，这是变更分类的基础
- 代码片段必须精确并包含文件路径，方便定位
- 对话中未涉及的内容，根据已有代码上下文推断补充
